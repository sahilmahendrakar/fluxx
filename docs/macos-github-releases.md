# macOS releases and GitHub auto-update feed

Fluxx's packaged macOS builds read update metadata from **`sahilmahendrakar/fluxx`** on GitHub Releases using `electron-updater`. Windows and Linux are unchanged and do not use this feed.

## Release artifact names

Electron Forge / `electron-packager` use `package.json` **`productName`** (`Fluxx`) for distributable basenames:

| Asset | Pattern | Examples (v0.1.2) |
|-------|---------|-------------------|
| DMG (first install) | `{productName}-{arch}.dmg` (stable across releases) | `Fluxx-arm64.dmg`, `Fluxx-x64.dmg` |
| Zip (auto-update) | `{productName}-darwin-{arch}-{version}.zip` | `Fluxx-darwin-arm64-0.1.2.zip`, `Fluxx-darwin-x64-0.1.2.zip` |
| Updater manifest | `latest-mac.yml` | lists both zip basenames and checksums; primary `path` is arm64 |

Per-arch DMG basenames come from `MakerDMG` in `forge.config.ts` (`darwinDmgBasename` in `src/build/macReleaseArtifacts.ts`). Latest install links:

- Apple Silicon: `https://github.com/sahilmahendrakar/fluxx/releases/latest/download/Fluxx-arm64.dmg`
- Intel: `https://github.com/sahilmahendrakar/fluxx/releases/latest/download/Fluxx-x64.dmg`

Legacy releases may still list a single `Fluxx.dmg`, versioned `Fluxx-*-arm64.dmg`, or `Flux-*` assets. Older **`sahilmahendrakar/fluxx-web`** release assets remain on that repo for historical installs; new tags publish to **`sahilmahendrakar/fluxx`**. Update [fluxx.sh](https://fluxx.sh) (separate **fluxx-web** repo if applicable) so Apple Silicon and Intel download CTAs point at the matching URLs above—not a shared `Fluxx.dmg`.

## Developer release flow

Use pnpm for version bumps because this repo is pnpm-managed and pins `packageManager` in `package.json`.

```bash
pnpm version patch
git push --follow-tags
```

Use `minor` or `major` instead of `patch` when appropriate. `pnpm version` updates `package.json`, creates the version commit, and creates the `vX.Y.Z` tag. Pushing the branch and tag with `git push --follow-tags` starts the GitHub Actions release workflow.

The workflow only runs for pushed tags that match `v*`, and it fails early if the tag name does not match the `package.json` version.

## CI release workflow

`.github/workflows/release.yml` runs on GitHub-hosted macOS runners and uses the existing Electron Forge pipeline:

```bash
pnpm install --frozen-lockfile
pnpm run publish:mac-dual
```

`publish:mac-dual` runs `electron-forge publish --arch=arm64,x64` (both Apple Silicon and Intel). Do **not** use `pnpm run publish -- --arch=arm64,x64`: pnpm forwards a literal `--` before `--arch`, so Forge ignores the arch flag and only builds for the host CPU (arm64 on Apple Silicon).

For local DMG/zip checks without uploading:

```bash
pnpm run make:mac-dual
```

`pnpm run publish` invokes the package script (`electron-forge publish`). Do not use bare `pnpm publish` in CI or locally for releases, because that is pnpm's npm-registry publish command rather than the package script.

After Forge uploads assets, CI sets the GitHub release **title** to `Fluxx X.Y.Z`, the **body** from `docs/github-release-notes-template.md` (substituting `{{VERSION}}`), and **`--draft=false`** so the release is public (`GITHUB_TOKEN` / Forge often leave new releases as drafts until this step). Edit that template before tagging when you want curated release notes instead of an empty Highlights section.

The workflow runs in **`sahilmahendrakar/fluxx`**, and Forge publishes to the same repository. The workflow grants the default `GITHUB_TOKEN` **Contents: write** and passes it into the publish step as `GITHUB_TOKEN` (required by `@electron-forge/publisher-github`). `gh release edit` uses the same token. No cross-repo PAT is required.

## Release assets

Forge publishes the app version to GitHub Releases through `@electron-forge/publisher-github`. The release is configured as a non-draft, non-prerelease release so `electron-updater` can see it as the latest stable version.

Every macOS release must include:

1. Two DMGs from `@electron-forge/maker-dmg` (`Fluxx-arm64.dmg`, `Fluxx-x64.dmg`) for first install.
2. Two zips from `@electron-forge/maker-zip` for Electron auto-updates (one per arch).
3. `latest-mac.yml`, generated by the Forge `postMake` hook next to the Darwin zips before publish uploads assets.

DMGs are for first install. The updater downloads and installs the zip for the running app’s architecture from `latest-mac.yml` (`files` lists both; `path` stays arm64 for backward compatibility).

## Required GitHub secrets

Add these repository secrets in **`sahilmahendrakar/fluxx`** under **Settings > Secrets and variables > Actions > Repository secrets**:

| Secret | Required for | Notes |
|--------|--------------|-------|
| `APPLE_ID` | Notarization | Apple Developer account email. |
| `APPLE_APP_SPECIFIC_PASSWORD` | Notarization | Preferred app-specific password secret. |
| `APPLE_PASSWORD` | Notarization | Legacy alias. The workflow maps `APPLE_APP_SPECIFIC_PASSWORD` or `APPLE_PASSWORD` into the `APPLE_PASSWORD` env var Forge has historically used, and `forge.config.ts` accepts either env var. |
| `APPLE_TEAM_ID` | Notarization | Apple Developer Team ID. |
| `APPLE_CERTIFICATE_BASE64` | Signing on GitHub-hosted macOS | Base64-encoded `.p12` Developer ID Application certificate. |
| `APPLE_CERTIFICATE_PASSWORD` | Signing on GitHub-hosted macOS | Password for the `.p12` certificate. Required when `APPLE_CERTIFICATE_BASE64` is set. |
| `APPLE_KEYCHAIN_PASSWORD` | Signing on GitHub-hosted macOS | Temporary CI keychain password. Required when `APPLE_CERTIFICATE_BASE64` is set. |

### App build secrets (same names as local `.env`)

Release builds inline these at compile time (renderer `VITE_*`, main process via `vite.main.config.ts`). Copy each value from your local `.env` into a GitHub secret with the **same name** — do not commit `.env`.

| Secret | Required for |
|--------|----------------|
| `VITE_FIREBASE_API_KEY` | Firebase auth + Firestore in packaged app |
| `VITE_FIREBASE_AUTH_DOMAIN` | Firebase |
| `VITE_FIREBASE_PROJECT_ID` | Firebase |
| `VITE_FIREBASE_STORAGE_BUCKET` | Firebase config parity with local `.env` |
| `VITE_FIREBASE_MESSAGING_SENDER_ID` | Firebase config parity with local `.env` |
| `VITE_FIREBASE_APP_ID` | Firebase |
| `VITE_GOOGLE_DESKTOP_CLIENT_ID` | Google sign-in (main process) |
| `VITE_GOOGLE_DESKTOP_CLIENT_SECRET` | Google sign-in (main process) |
| `RESEND_API_KEY` | Invite / transactional email (main process) |
| `RESEND_FROM_DOMAIN` | Resend From domain |
| `RESEND_FROM_NAME` | Resend From display name |

Do not configure `CSC_LINK` or `CSC_KEY_PASSWORD` for this workflow. Those are electron-builder-style secrets, and this Forge/electron-packager signing path does not read them. The workflow imports the Developer ID certificate into a temporary macOS keychain so `osxSign: {}` can discover it normally.

### Deprecated: `RELEASE_GITHUB_TOKEN`

Previously, CI ran in this repo but published releases to **`sahilmahendrakar/fluxx-web`**, which required a `RELEASE_GITHUB_TOKEN` secret (PAT with `contents: write` on `fluxx-web`). That secret is **no longer used**. You can remove it from repository secrets after confirming a tag-driven release succeeds with the default `GITHUB_TOKEN`.

If you ever need a PAT again (for example, publishing from a different workflow repo), it must target **`sahilmahendrakar/fluxx`** with **Contents: read and write**, not `fluxx-web`.

## Manual GitHub setup

1. In **`sahilmahendrakar/fluxx`**, open **Settings > Secrets and variables > Actions > Repository secrets** and add `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD` (or legacy `APPLE_PASSWORD`), `APPLE_TEAM_ID`, `APPLE_CERTIFICATE_BASE64`, `APPLE_CERTIFICATE_PASSWORD`, and `APPLE_KEYCHAIN_PASSWORD`.
2. Make sure GitHub Actions are enabled for `sahilmahendrakar/fluxx`. The release workflow needs **Contents: write** on the default `GITHUB_TOKEN` (configured in `.github/workflows/release.yml`).
3. Create the next release by running `pnpm version patch` (or `minor` / `major`) and `git push --follow-tags`.
4. Confirm the tag workflow succeeds and the matching **`sahilmahendrakar/fluxx`** GitHub Release contains `Fluxx-arm64.dmg`, `Fluxx-x64.dmg`, both versioned zips, and `latest-mac.yml`, with title **Fluxx X.Y.Z**.

## App runtime

- `autoUpdater.autoDownload` is `false`; no installer is fetched unless the app calls `downloadUpdate()`.
- `FLUX_DISABLE_GITHUB_UPDATES=1` turns off configuring the updater and rejects update checks via IPC.

To verify the feed from an installed packaged macOS build, use the app's update check path. It calls `electron-updater` against the GitHub provider configured for `sahilmahendrakar/fluxx`, reads the latest non-draft GitHub Release, downloads `latest-mac.yml`, and advertises a newer semver when one is available.

## Private repos

Public `fluxx` needs no GitHub auth for clients to read release assets. For a private releases repo later, configure `electron-updater` auth (token / `addAuthHeader`) in addition to publisher credentials. That is not required for the public open-source repo.
